# AI Briefing — chief-of-staff dashboard

The Quest dashboard opens with a chief-of-staff briefing: a greeting, a
one-sentence "why today looks like this" context line, 1–3 task picks with
reasoning, and a signoff. The briefing is what the user approves, swaps,
or adjusts — they should never see an empty "tap to set priorities" form.

## Endpoints

- `POST /api/briefing/today` — returns today's briefing (cached for 1 hour
  per user/date in the `daily_briefings` table). Re-fetches generate the
  same response unless cache is bypassed.
- `POST /api/briefing/reshuffle` — bypasses cache and regenerates with a
  "give me different suggestions" hint.
- `POST /api/briefing/approve` — marks today's briefing as accepted and
  surfaces a "Plan locked in" badge.

Response shape:

```ts
type BriefingResponse = {
  greeting: string;     // "Good morning, Theresa."
  headline: string;     // "Three things matter today."
  context: string;      // 1-sentence narration of *why*
  briefing: Array<{
    taskId: string | number | null;  // existing task id, or null for an AI suggestion
    title: string;
    pillarName: string;
    pillarColor: string | null;
    priority: "P1" | "P2" | "P3" | "P4";
    reasoning: string;             // "Surfaced because..."
    estimatedMinutes: number;
    suggestedNextStep: string | null;
    blockedBy: string | null;
  }>;
  signoff: string;
  date: string;
  source: "ai" | "rules" | "fallback";
  approved: boolean;
  generatedAt: string;
};
```

## AI mode (gpt-4o-mini)

When `OPENAI_API_KEY` is set, the server calls
`POST https://api.openai.com/v1/chat/completions` with:

- `model: "gpt-4o-mini"`
- `response_format: { type: "json_object" }`
- A system prompt establishing the chief-of-staff persona and the JSON
  schema the model must return.
- A user prompt with the user's pillars, weekly priorities, open tasks,
  recently completed tasks, and the local time of day.

The response is parsed, validated against the user's actual task IDs (any
made-up ID is set to `null`), and returned with `source: "ai"`.

A request that times out (default 8s) or fails for any reason falls back
to the rules-based briefing with `source: "fallback"`. The cached row in
`daily_briefings` records which source produced the briefing.

### Cost estimate

The user prompt is small (a few hundred tokens of context) and the
response is constrained to a small JSON object. With gpt-4o-mini at the
April 2026 price (~$0.15 / 1M input, $0.60 / 1M output), one briefing
costs roughly **$0.0001**. The 1-hour cache means a typical user
generates 1–3 briefings/day even with refresh.

## Rules-based mode (default)

When `OPENAI_API_KEY` is not set, the briefing is generated by
`buildRulesBriefing` (see `artifacts/api-server/src/lib/briefing/rules.ts`).
The picker:

1. Collects open tasks (`pending` or `blocked`).
2. Sorts by pillar priority (P1 > P2 > P3 > P4), then by whether the
   pillar is on the active list this week, then recency.
3. Caps the result at 3 items.
4. Builds a context line that references the most recent completion and
   the P1 active pillar by name.

Greeting and signoff use templated copy with the user's first name and
local hour.

## Caching

`daily_briefings` is a single source of truth keyed on `(user_id, date)`.
The row stores the JSON briefing, when it was generated, when (if at all)
it was approved, and which source produced it. The picker re-uses the row
if it was generated within the last hour; otherwise it generates a new
one and overwrites the row.

`POST /api/briefing/reshuffle` bypasses the freshness check and
regenerates immediately, and `POST /api/briefing/approve` only stamps
`approved_at` — it does not overwrite the JSON.

## Owner-mode + `req.userId`

The routes use `getUserId(req)` to read the request-scoped user id. When
no scope is set (single-tenant mode), `userId` is `null` and the unique
index treats `(NULL, date)` as a single row.
